# CONTROLS.md — Compliance Control Matrix

> The 15-control matrix mapping every technical control in this architecture to its PHIPA, PIPEDA, and Quebec Law 25 obligations, with the Terraform file (or runbook) that implements it and an interview-ready one-liner for each row.
>
> **Reading guide.** `ID` is the control number (C1..C15). `Implementation` is the Terraform resource + module that enforces the control. `PHIPA`, `PIPEDA`, and `Law 25` cite the specific clause / principle / article the control satisfies. `Evidence` is a live link to the file in this repo that proves the control is implemented (or to a runbook that walks the control in action). `One-liner` is a 1-sentence, interview-ready answer to "how do you enforce `[X]`?"
>
> The matrix is the project's primary deliverable: AWS publishes a HIPAA reference architecture but nothing for PHIPA or Law 25. This matrix does the Canadian-framework cross-walk that AWS itself has not.

## Glossary (acronyms used in the matrix)

| Acronym | Meaning |
|---------|---------|
| **PHIPA** | Personal Health Information Protection Act, 2004 (Ontario) — `S.O. 2004, c. 3, Sched. A` |
| **PIPEDA** | Personal Information Protection and Electronic Documents Act (Canada, federal) |
| **Law 25** | Quebec's *Act to modernize legislative provisions as regards the protection of personal information* (*Loi 25*), in force since September 2022, fully applicable since September 2024 |
| **IPC** | Information and Privacy Commissioner of Ontario (the PHIPA regulator) |
| **OPC** | Office of the Privacy Commissioner of Canada (the PIPEDA regulator) |
| **CAI** | *Commission d'accès à l'information* du Québec (the Quebec Law 25 regulator) |
| **RROSH** | Real Risk of Significant Harm — the PIPEDA test for whether a breach is reportable |
| **PHI** | Personal Health Information (PHIPA's term for health data) |
| **MTTD** | Mean Time To Detect |
| **CMK** | Customer-Managed Key (AWS KMS) |
| **IdC** | AWS IAM Identity Center (formerly AWS SSO) |
| **OIDC** | OpenID Connect (used for CI/CD → AWS role assumption) |
| **WORM** | Write Once, Read Many — the storage property Object Lock COMPLIANCE provides |

## How to read this matrix

1. **Pick a row by clause (audit walkthrough).** An auditor asking "how do you satisfy PHIPA s.13?" starts at column `PHIPA`, scans down, and lands on **C1, C2, C3, C4, C5, C7, C10, C14, C15** — nine controls, each with a single evidence link to a file in this repo. The auditor can `cat` each file and see the control in code.
2. **Pick a row by resource (security review).** A security engineer asking "is the S3 PHI bucket properly hardened?" lands on **C1** (the CMK), **C2** (TLS), **C3** (IAM scope), **C4** (network), **C5** (audit), **C6** (Config), **C7** (GuardDuty), **C9** (residency), **C10** (CRR), **C11** (rotation) — ten controls reference the bucket. No single control is "the" answer; the controls are layered.
3. **Pick a row by failure mode (incident response).** "An RDS snapshot got copied to a public bucket — what now?" lands on **C15** (the runbook), with the T0 detection at the **C7 + Macie** line, the T+2h containment at the **C1 + C3** lines (rotate the CMK + the service role), and the T+72h notification at the **C15** regulator clauses (PHIPA s.13 / PIPEDA Sch.1 / Law 25 §3.1).

## What this matrix is NOT

- It is **not legal advice**. The cited clauses are accurate to the best of the author's reading at 2026-06-23 and may be incomplete or out of date. A real HealthTech would have outside counsel review this matrix before any regulatory submission.
- It is **not a SOC 2 / HITRUST CSF cross-walk**. SOC 2 + HITRUST CSF v11 are explicitly deferred to follow-up work (see the plan's "Deferred to Follow-Up Work" section).
- It is **not a HIPAA mapping**. The Canadian frameworks are the requirement; HIPAA is a US framework and is out of scope for this project.
- It is **not exhaustive**. PHIPA + PIPEDA + Law 25 + the O. Reg. 329/04 safeguards regulation have more granular obligations (e.g., specific breach notification timelines to the IPC vs the OPC vs the CAI) that a production privacy program would track separately.

## What it is

A 15-control cross-walk that an interviewer can read in 10 minutes and a regulator can audit by `cat`-ing one file per control. Every control is implemented in the repo, every evidence link resolves to a real file, and every interview one-liner is the kind of answer you would give in a 30-minute technical round at a Canadian HealthTech.
